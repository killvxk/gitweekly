# 描述生成规则

## 基本要求

生成的中文描述必须符合以下规范：

### 长度
- **15-25 字**（不含标点符号）
- 过短：信息不足，无法理解项目
- 过长：冗余，影响阅读体验

### 格式模板

推荐使用以下格式之一：

1. **`<核心功能>的<技术栈/领域>工具/库/框架`**
   - 示例："专为Windows 7系统定制的Go语言编译器"

2. **`<技术栈>实现的<核心功能>解决方案`**
   - 示例："基于Unicorn引擎的Windows二进制逆向调试仿真混合工具"

3. **`<领域>的<核心功能>平台/系统`**
   - 示例:"AI代理群自动化挖掘Windows内核驱动漏洞的大规模安全研究"

### 内容要求

1. **突出核心功能**
   - 说明项目"做什么"，而非"是什么"
   - 避免空洞词汇："强大的"、"优秀的"、"完善的"

2. **保留专有名词**
   - 技术栈名称保留英文：Go、Python、React、Windows
   - 产品名称保留原文：GitHub、Docker、Kubernetes

3. **简洁明了**
   - 一句话说清楚
   - 避免从句和修饰语堆砌
   - 不使用"一个"、"这是"等冗余词

## 生成步骤

### 1. 提取关键信息

从 WebFetch 结果中提取：
- **项目类型**：库/框架/工具/平台/系统
- **核心功能**：做什么（动词短语）
- **技术栈**：使用什么技术
- **特色**：与同类项目的区别

### 2. 组织描述结构

按优先级选择信息：
1. 核心功能（必须）
2. 技术栈（推荐）
3. 特色/优势（可选）
4. 应用场景（可选）

### 3. 精简表达

- 删除冗余词："一个"、"这是"、"用于"
- 合并相似概念："高性能且可扩展" → "高性能可扩展"
- 使用专业术语：避免口语化表达

### 4. 验证质量

检查清单：
- [ ] 长度在 15-25 字
- [ ] 说明了核心功能
- [ ] 没有空洞词汇
- [ ] 专有名词正确
- [ ] 一句话说清楚

## 特殊情况处理

### GitHub 仓库

**README 内容充足**:
- 提取 README 中的项目描述
- 关注 "About"、"Features"、"What is" 部分

**README 内容不足**:
- 使用仓库描述（GitHub description）
- 结合文件结构推断（如 package.json、setup.py）

### 非 GitHub URL

**技术文档**:
- 格式："<产品名> <文档类型>，<核心内容>"
- 示例："Intel VTune性能分析工具的ITT API集成指南"

**博客文章**:
- 格式："<主题>的<技术/方法>实战/研究"
- 示例："7天构建AI语音钓鱼系统的红队社会工程自动化实战"

**工具网站**:
- 格式："<功能>的<技术>可视化/分析工具"
- 示例:"Windows System32二进制文件依赖关系交互式可视化分析工具"

### 失效 URL

如果 WebFetch 失败（404、超时等）：
1. 标记为 `__DELETED__`
2. 写入缓存
3. 跳过该 URL

## 领域特定规则

### 安全/渗透测试

强调：
- 攻击/防御类型
- 目标系统
- 自动化程度

示例：
- "AI代理群自动化挖掘Windows内核驱动漏洞的大规模安全研究"
- "集成AI辅助的下一代模块化C2渗透测试框架"

### AI/机器学习

强调：
- 模型类型
- 优化目标
- 应用场景

示例：
- "专为Apple芯片优化的脉冲神经网络库"
- "AI驱动的多平台研究工具，聚合分析8个社交平台最近30天热门内容"

### 开发工具

强调：
- 目标平台/语言
- 核心功能
- 集成方式

示例：
- "Unity Editor的CLI工具，让AI代理通过命令行控制编辑器"
- "将自然语言创意转化为Seedance 2.0视频提示词的Claude技能"

## 质量评分标准

### 优秀（9-10分）
- 长度适中（18-22字）
- 核心功能清晰
- 技术栈明确
- 特色突出
- 无冗余词

### 良好（7-8分）
- 长度合格（15-25字）
- 核心功能清晰
- 技术栈或特色之一明确
- 少量冗余词

### 及格（5-6分）
- 长度合格
- 核心功能基本清晰
- 有一定信息量

### 不及格（<5分）
- 长度不合格
- 核心功能不清
- 空洞或重复 URL 信息
- 冗余词过多

## 常见错误

### 1. 重复 URL 信息
❌ "GitHub上的开源项目"
✓ "开源的XXX工具"

### 2. 使用空洞词汇
❌ "强大的数据处理工具"
✓ "高性能分布式数据处理引擎"

### 3. 过度详细
❌ "基于Python 3.8+开发的Web应用框架，支持异步I/O、RESTful API、WebSocket、数据库ORM"
✓ "异步Python Web框架，支持RESTful和WebSocket"

### 4. 主观评价
❌ "非常优秀的机器学习库"
✓ "轻量级机器学习库，专注于边缘设备部署"

### 5. 冗余词堆砌
❌ "这是一个用于帮助开发者进行代码审查的工具"
✓ "代码审查自动化工具"
